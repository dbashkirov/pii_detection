# Синтетический датасет: генерация

## Мотивация

Для создания модели детекции русскоязычных ПД наиболее проблемными классами являются **NAME** и **ADDRESS**. 
Их вариативность делает регулярки неприменимыми, а существующие модели либо слишком тяжёлые, 
либо недостаточно хорошо справляются с выделением сущностей полностью и во всем их разнообразии.

Решение — дообучить компактную NER-модель именно на NAME и ADDRESS. 
Для этого нужен размеченный датасет с разнообразным контекстом (доставка, банк, юридический и т.д.).
Имеющиеся в открытом доступе датасеты либо слишком малы, либо слишком узкие по домену, либо не покрывают все разнообразие сущностей.

Датасет был сгенерирован синтетически с помощью Claude.

## Архитектура пайплайна

Все файлы находятся в папке `data_gen/`:

| Файл | Назначение |
|------|-----------|
| `sample_next.py` | Сэмплирует случайную спецификацию для одного примера, выводит JSON |
| `generation_prompt.md` | Инструкция для Claude: правила генерации текста по спецификации |
| `_gen_batch.py` | Вспомогательный скрипт для батчевой записи |
| `output/synthetic_pii.jsonl` | Мастер-файл датасета |

### sample_next.py

Скрипт полностью контролирует баланс датасета. При каждом вызове он сэмплирует с заданными вероятностями:

Подобный процесс позволяет разнообразить датасет и, насколько это возможно, предотвратить постоянные повторения.

- **domain** — домен примера (13 доменов): DELIVERY, DIALOG, AUTO, BANK, HR, RE, TELECOM, NAME, ADDRESS, MEDICAL, GOV, LEGAL, SOCIAL. У каждого домена своя целевая квота (500–1000 примеров).
- **entity_type** — что аннотировать: NAME_ONLY (20%), ADDRESS_ONLY (20%), BOTH (30%), NEGATIVE (25%, без ПД), NAME_CONTEXT (5%, имя + упоминание телефона/email без аннотации).
- **name_form** — форма имени (12 вариантов): от NAME_FI (Имя Фамилия, 25%) до склонённых форм NAME_FIO_GEN/NAME_FIO_DAT (по 1%). Русские падежи обрабатываются через `pymorphy3`.
- **addr_form** — формат адреса (10 вариантов): от ADDR_FULL (полный адрес, 20%) до ADDR_INDEX (только индекс, 2%).
- **style** — стиль текста: formal / informal / mixed.
- **length** — длина: short (1 предложение), medium (2–4), long (4+).
- **expression** — эмоциональный тон: neutral, joy, irritated, sad, urgent, worried, rage.
- **edge_case** — граничный случай (20% позитивных примеров): `hyphenated_surname`, `non_slavic_name`, `complex_house_number`, `lowercase_city`, `initials_nospace` и др.
- **neg_category** — категория негативного примера: `professions_roles`, `city_homonyms`, `memorial_streets`, `email_names`, `org_names` и др.
- **has_typo** (5%) — одна опечатка в контексте, но не в самой сущности.
- **multi_entity** (12%) — несколько ПД в одном тексте.
- **position** — позиция сущности в тексте: beginning / middle / end / mixed.

Скрипт также генерирует конкретные данные для вставки через Faker `ru_RU`.

### generation_prompt.md

Подробная инструкция, которую получает Claude вместе со спецификацией. Описывает:
- правила сборки имени для каждой из 12 форм (включая склонение);
- требование вставлять `address_string` дословно, без переформулировки;
- правило: опечатки только в контексте, не в тексте сущности;
- формат диалогов (DIALOG — текст в виде JSON-строки с сообщениями);
- правило эмоционального тона (rage → 1–3 ненормативных слова в контексте);
- гарантию: текст сущности должен быть точной подстрокой поля `text`.

## Процесс генерации

Генерация велась итеративным loop'ом через **Claude Code**:

1. Запустить `python data_gen/sample_next.py` → получить JSON-спецификацию одного примера.
2. Claude читает спецификацию, следует `generation_prompt.md`, генерирует текст.
3. Результат добавляется к `output/synthetic_pii.jsonl`.

Для ускорения использовался **батчевый режим**: Claude Code редактировал `_gen_batch.py`, хардкодя в него сразу несколько спецификаций с готовыми текстами и вызывая скрипт для записи. Файл `_gen_batch.py` — вспомогательный, он не является частью пайплайна, а служил инструментом Claude Code для групповой вставки примеров.

## Формат записи

Каждая строка в `synthetic_pii.jsonl` — JSON-объект:

```json
{
  "id": "del_synth_0042",
  "domain": "DELIVERY",
  "entity_type": "BOTH",
  "name_form": "NAME_FI",
  "addr_form": "ADDR_CSH",
  "style": "informal",
  "length": "medium",
  "expression": "irritated",
  "neg_category": null,
  "edge_case": null,
  "has_typo": false,
  "multi_entity": false,
  "position": "middle",
  "text": "Курьер должен передать посылку Алексею Петрову по адресу ул. Садовая, д. 5, кв. 12.",
  "entities": [
    {"text": "Алексею Петрову", "type": "NAME"},
    {"text": "ул. Садовая, д. 5, кв. 12", "type": "ADDRESS"}
  ]
}
```

Офсеты не хранятся — только текст сущности, который гарантированно является точной подстрокой `text`.

## Объём и целевое распределение

Сгенерировано 3 030 примеров.

Целевое распределение по типу сущности:
- NAME only — 20%
- ADDRESS only — 20%
- BOTH — 30%
- NEGATIVE — 25%
- NAME_CONTEXT — 5%

### Недостатки

Помимо классических проблем синтетических данных, существуют следующие недостатки:

1. Аспекты expression и position добавлялись не с самого начала генерации и заполнены не у всех примеров.
2. Из-за изменения процесса генерации edge-case non_slavic_name не имеет смысла, т.к. имена приходят от faker и случайны, а не генерируются синтетически.
3. Аббревиатуры городов получаются простым сокращением и часто нереалистичны.
