# PII Detector

Детектор персональных данных для русскоязычного текста. 
Гибридная архитектура: дообученная spaCy NER-модель для имён и адресов, регулярные выражения 
с контрольными суммами для структурированных идентификаторов, энтропийный анализ для токенов. 
Построен на [Microsoft Presidio](https://github.com/microsoft/presidio).

Предназначен для встраивания в LLM-прокси или любой сервис, 
где нужно перехватывать персональные данные до отправки во внешние системы.

---

## Детектируемые типы

| Категория                      | Типы                                                       |
|--------------------------------|------------------------------------------------------------|
| Персональные                   | `NAME`, `ADDRESS`                                          |
| Контакты                       | `PHONE_NUMBER`, `EMAIL`                                    |
| Банковские                     | `BANK_CARD_NUMBER`, `CVC`                                  |
| Государственные идентификаторы | `INN`, `SNILS`, `OGRN`, `OGRNIP`, `KPP`, `PASSPORT_NUMBER` |
| Секреты / токены               | `TOKEN`                                                    |

---

## Архитектура


**spaCy NER** — дообученная модель `ru_core_news_lg` с NER-головой на два класса. 
Покрывает `NAME` и `ADDRESS`, так как их вариативность делает регулярки неприменимыми. 
Подробности дообучения и сравнение трёх конфигураций — в [`docs/finetune_spacy.md`](docs/finetune_spacy.md).

**Regex + контрольные суммы** — каждый паттерн (где применимо) работает совместно с математической валидацией 
(алгоритм Луна для карт, контрольные суммы ФНС для ИНН/ОГРН/СНИЛС) и контекстным окном ±60 символов. 
Контекст влияет на уверенность.

**Энтропийный детектор** — обнаруживает токены двумя способами: 
конкретные форматы (JWT, GitHub PAT, OpenAI/Anthropic ключи, AWS, Stripe, Telegram и др.) и 
высокоэнтропийные строки с порогом, скорректированным на наличие контекстных ключевых слов. 
Дополнительно ловит OTP-коды рядом с соответствующими триггерами.

**Дедупликация** — перекрывающиеся спаны разрешаются по правилам: 
совпадение позиции с разными типами → оставить с наивысшим score; 
совпадение типа с разной длиной → оставить более длинный спан; 
иные пересечения → оставить с наивысшим score.

## Демо-стенд

Локальный LLM-стек с PII-фильтрацией в диалоге: Open WebUI → LiteLLM (нативный Presidio guardrail) → Ollama. Все запросы остаются на машине. Политика — `MASK` (ПД заменяются плейсхолдерами до LLM, обратно подставляются исходные) или `BLOCK` (HTTP 400).

```bash
cp docker/.env.example docker/.env  # заполнить секреты по примеру
docker compose -f docker/docker-compose.yml --env-file docker/.env up --build -d
```

После запуска: Open WebUI — http://localhost:3000, LiteLLM UI — http://localhost:4000/ui

Подробнее — в [`docker/README.md`](docker/README.md).

## Использование

### Установка

```bash
pip install presidio-analyzer presidio-anonymizer spacy phonenumbers pymorphy3
```

### Детекция и маскирование

```python
from pii_detector import HybridPIIDetector

detector = HybridPIIDetector(spacy_model_path="training/models/model-best-combo-0.78")

# Детекция
text = "Иван Петров, ИНН 500100732259, тел. +7 916 123 45 67"
results = detector.analyze(text)
for r in results:
    print(r.entity_type, repr(text[r.start:r.end]), r.score)
# NAME      'Иван Петров'      0.85
# INN       '500100732259'     0.90
# PHONE_NUMBER '+7 916 123 45 67' 0.95

# Маскирование
print(detector.anonymize("Email: test@example.com, СНИЛС 112-233-445 95"))
# Email: <EMAIL>, СНИЛС <SNILS>
```

Больше примеров — в [`notebooks/demo.ipynb`](notebooks/demo.ipynb).

**Примечания по классам с заметными ошибками:**

- ***BANK_CARD_NUMBER** — В исходном датасете номера карт генерировались с некорректными контрольными суммами. 
Данные отредактированы, чтобы последняя цифра номера карты была верной по алгоритму Луна.
- **CVC** — три FN: рядом с кодом нет контекстного слова (CVV/CVC/код карты). Намеренно не детектируем любые три цифры без контекста.
- **EMAIL** — один FN: адрес написан кириллицей, такой формат почти не встречается в реальных данных, шаблон не добавлялся.
- **INN / ОГРНИП** — по два FN каждый: неверная контрольная сумма, то есть невалидны.
- **PASSPORT_NUMBER** — три FN: в тексте только серия или только номер без второй части, неполный формат намеренно не покрывается. Один FN — невалидный номер. Один FP — валидный номер, но по контексту датасета это «пример» номера, а не реальные данные.
- **PHONE_NUMBER** — один FN: невалидный номер. Один FP — аналогично паспорту: формально валидный номер, не размеченный в датасете из-за контекста.

## Синтетический датасет

Для дообучения NER не существует публичного русскоязычного датасета с классами NAME и ADDRESS в сервисном тексте. 
Датасет (~3 000 примеров, 13 доменов) был сгенерирован с помощью Claude через итеративный loop: 
скрипт `training/data_gen/sample_next.py` сэмплирует спецификацию примера, Claude генерирует текст по инструкции. 
Подробнее — в [`docs/synth_generation.md`](docs/synth_generation.md).
