Ты генерируешь примеры для PII-датасета (NAME + ADDRESS, рус.). Каждая итерация — один пример.

## Workflow итерации

```
1. python data_gen/sample_next.py                                  ← получи спецификацию
2. wc -l data_gen/output/synthetic_pii.jsonl                       ← получи N (кол-во строк = кол-во записей)
3. id = f"{id_prefix}_{N+1:04d}"                                   ← следующий ID
4. Сгенерируй один пример по спеку                                 ← см. правила ниже
5. Добавь одну JSON-строку в КОНЕЦ файла (не читай файл целиком)
6. Повтори
```

**ID:** формат `{id_prefix}_{N+1:04d}` — глобальный счётчик через строки файла. Например, если в файле 47 строк, следующий ID = `del_synth_0048` (или `bank_synth_0048` — зависит от id_prefix спека).

**Запись в файл** — добавляй в конец без чтения всего файла:
```bash
echo '{...json...}' >> data_gen/output/synthetic_pii.jsonl
```

## Формат выходной записи

Одна JSON-строка (без переноса строк внутри). Все поля из спека переносятся дословно.

```json
{"id": "del_synth_0048", "domain": "DELIVERY", "entity_type": "BOTH", "name_form": "NAME_FI", "addr_form": "ADDR_SHA", "style": "informal", "length": "medium", "expression": "neutral", "neg_category": null, "edge_case": null, "has_typo": false, "multi_entity": false, "position": "end", "text": "Жду курьера, уже час прошёл. Привезите на ул. Луначарского 156 кв. 89, получатель Антон Кузнецов", "entities": [{"text": "ул. Луначарского 156 кв. 89", "type": "ADDRESS"}, {"text": "Антон Кузнецов", "type": "NAME"}]}
```

- `entities` — только `text` и `type`. **Без `start`/`end`** — офсеты вычислит скрипт конвертации по вхождению строки в `text`.
- Текст в `entities[i].text` должен буквально совпадать с подстрокой `text` — никаких изменений, точно то же самое.
- `entities` = `[]` для негативных примеров
- Телефоны и email в тексте не аннотируются

## Как использовать name_data и addr_data

### Адрес

`addr_data.address_string` — вставлять **дословно** в текст как адрес. Это уже готовая строка в нужном формате.

### Имя

Собирать из компонентов `name_data` согласно `name_form`:

| name_form | Как собирать |
|-----------|-------------|
| `NAME_FI` | `{firstname} {surname}` |
| `NAME_IF` | `{surname} {firstname}` |
| `NAME_FIO` | `{firstname} {patronymic} {surname}` |
| `NAME_IFO` | `{surname} {firstname} {patronymic}` |
| `NAME_IO` | `{firstname} {patronymic}` |
| `NAME_I` | `{firstname}` |
| `NAME_INITF` | `{firstname[0]}.{patronymic[0]}. {surname}` |
| `NAME_FINIT` | `{surname} {firstname[0]}.{patronymic[0]}.` |
| `NAME_FI_GEN` | `{firstname} {surname}` в родительном падеже |
| `NAME_IF_GEN` | `{surname} {firstname}` в родительном падеже |
| `NAME_FIO_GEN` | `{surname} {firstname} {patronymic}` в родительном |
| `NAME_FIO_DAT` | `{surname} {firstname} {patronymic}` в дательном |

Склонение косвенных падежей — применяй знание русской грамматики. Поле `full_fio` (именит.) дано для справки.

### multi_entity: true

Спек содержит `name_data_2` и/или `addr_data_2`. Оба спана должны попасть в текст и в `entities`.  
Текст должен органично упоминать оба — например, две стороны сделки, два адреса доставки.

### has_typo: true

Ввести одну опечатку в произвольное слово **контекста** (не в сущность). Например: "достаивть", "курьерр", "добый день".

## Правила генерации

### position

Указывает, где сущность(и) располагаются в тексте.

| position    | Смысл |
|-------------|-------|
| `beginning` | Сущность в первой трети текста |
| `middle`    | Сущность примерно в середине (не в первом и не в последнем предложении при medium/long) |
| `end`       | Сущность в последней трети; до неё — контекст без PII |
| `mixed`     | Сущность упоминается несколько раз **или** несколько сущностей расположены в разных частях текста |
| `null`      | Только для NEGATIVE — поле игнорируется |

При `length: short` (одно предложение): `beginning` = начало фразы, `end` = конец, `middle` = середина.  
При `multi_entity: true` и `position` = `beginning`/`middle`/`end` — оба спана в этой зоне; при `mixed` — они должны быть в разных частях текста.

---

### style

- `formal` — официальный язык, полные слова, «Уважаемые...», знаки препинания
- `informal` — мессенджер-стиль, сокращения «пж» / «спс» / «норм», неполные предложения. Нецензурная лексика управляется полем `expression` (см. ниже), не стилем.
- `mixed` — деловой нейтральный

### expression

Поле задаёт эмоциональный тон всего текста. Применяется ко всем типам примеров, включая NEGATIVE.

| expression  | Тон                             | Нецензурная лексика                                                 |
|-------------|---------------------------------|---------------------------------------------------------------------|
| `joy`       | Радость, восторг, благодарность | Допускается, но не требуется                                        |
| `neutral`   | Нейтральный, деловой            | Допускается, но не требуется                                                                 |
| `irritated` | Раздражение, недовольство       | Допускается, но не требуется                                                                 |
| `sad`       | Грусть, расстроенность, обида   | Допускается, но не требуется                                                                 |
| `urgent`    | Срочность, паника, спешка       | Допускается, но не требуется                                                                 |
| `worried`   | Тревога, беспокойство           | Допускается, но не требуется                                                                 |
| `rage`      | Ярость                          | Обязательно 1–3 слова мата в контексте (не внутри спанов сущностей) |

**Важно:**
- При `rage` мат — обязательное условие, без него пример не соответствует спеку. Используй весь спектр русского мата, а не только базовые выражения. Разрешено использовать оскорбления т.к. они часто встречаются в реальных данных.
- `expression` ортогонален `style`: `rage` может быть при любом стиле (официальная жалоба с матом — тоже реальный случай), но наиболее органично при `informal`.
- Нецензурные слова никогда не входят в спан сущности — только в окружающий контекст.

### length

- `short` — одно предложение, телеграфный стиль
- `medium` — 2–4 предложения с контекстом
- `long` — 4+ предложений; сущность не обязана быть в первом предложении

### entity_type

- **NAME_ONLY** — только NAME в entities, ADDRESS нет
- **ADDRESS_ONLY** — только ADDRESS в entities, NAME нет
- **BOTH** — обе сущности
- **NEGATIVE** — `entities: []`, никаких PII в тексте
- **NAME_CONTEXT** — NAME есть, в тексте словами упоминается телефон/email (не аннотируем)

### neg_category (только для NEGATIVE)

| neg_category | Как писать |
|--------------|-----------|
| `professions_roles` | Роли «мастер», «курьер», «риелтор» — без имени человека |
| `city_homonyms` | Топоним-омоним: «поехали во Владимир», «живёт в Туле» |
| `memorial_streets` | Улица имени человека без самого человека: «дом на ул. Пушкина» |
| `address_fragments` | Частичный адрес без улицы: «5 этаж», «корпус 3» |
| `email_names` | Имя внутри email: «напишите ivan@mail.ru» |
| `neutral_service` | «оплата прошла», «заказ подтверждён», «соединяю с оператором» |
| `org_names` | Имя в названии юрлица: «ООО Иванов и партнёры» |

### edge_case (применить обязательно, если не null)

**NAME edge cases:**
- `hyphenated_surname` — двойная фамилия уже в `surname`: использовать как есть, не забыть вставить в разметку все части, если они разделены
- `non_slavic_name` — нестандартное имя уже в `firstname`: использовать как есть
- `name_with_title` — добавить должность/роль рядом с именем: «директор Иванов», «пациент Маслова М.Г.»
- `initials_no_space` — инициалы слитно: И.И.Иванов (без пробела между инициалами и фамилией)
- `name_in_brackets` — ФИО в скобках: (Петров Александр Иванович) - в разметке без скобок, в тексте с ними
- `declined_full_fio` — ФИО в косвенном падеже: «Иванова Ивана Ивановича», «Ивановой Анне»

**ADDRESS edge cases:** уже применены в `address_string`, просто вставить дословно.

## DIALOG: особый формат

Домен DIALOG — `text` является JSON-строкой чата. Офсеты по ней целиком включая экранирование.

```
text = "[{\"role\": \"user\", \"content\": \"доставьте на ул. Ленина 15\"}, {\"role\": \"assistant\", \"content\": \"принято\"}]"
```

## Домены

| Домен | Формат | Особенности |
|-------|--------|-------------|
| DELIVERY | plain | Курьер/доставка. Много NAME+ADDRESS. Разговорный и смешанный стиль. |
| DIALOG | JSON-строка чата | ЖКХ, такси, доставка. Офсеты по всей JSON-строке. |
| AUTO | plain | ОСАГО, автосервис, ГИБДД. NAME = владелец ТС, ADDRESS = прописка или парковка. |
| BANK | plain | Банк, кредиты, верификация. Преобладает формальный. NAME в подписях и заявках. |
| HR | plain | Резюме, трудовые договоры. Строго формальный. Полное ФИО (NAME_IFO) стандарт. |
| RE | plain | Аренда и продажа недвижимости. ADDRESS — главная сущность. |
| TELECOM | plain | Заявки на подключение, техподдержка. NAME + ADDRESS абонента. |
| NAME | plain | Изолированные имена. entity_type всегда NAME_ONLY. |
| ADDRESS | plain | Изолированные адреса. entity_type всегда ADDRESS_ONLY. |
| MEDICAL | plain | Запись к врачу, медкарты, рецепты. Формальный. NAME_IFO как пациент. |
| GOV | plain | МФЦ, налоговая, суды. Строго формальный. Канцеляризмы. Полное ФИО + полный адрес. |
| LEGAL | plain | Договоры, нотариат, суды. «Проживающий по адресу», «именуемый в дальнейшем», «гражданин». |
| SOCIAL | plain | Переписка, посты, комментарии. Максимально разговорный. Опечатки, эмодзи, сленг. |

## Дополнительные указания

1. Не повторяй один и тот же тип действия (запрос / жалоба / уточнение / подтверждение) два раза подряд.
2. Разрешены бытовые, неидеальные формулировки: обрывки фраз, разговорные конструкции, неполные предложения.
3. Структура предложения должна отличаться от предыдущего примера хотя бы по одному параметру:
- порядок слов
- длина первого предложения
- наличие/отсутствие обращения
4. Не использовать одинаковое обращение чаще чем 1 раз подряд. (Привет, добрый день, здравствуйте, уважаемые, коллеги, друзья, ребята, господа, товарищи, etc.)
5. Каждый пример должен звучать как сообщение от другого человека в другой ситуации.


## Самопроверка перед записью

1. `entities[i].text` буквально входит в `text` как подстрока — не изменён ни один символ
2. `address_string` вставлен дословно — не переформулирован
3. `edge_case` применён (если не null)
4. `has_typo` — опечатка в контексте (не в сущности)
5. `multi_entity` — оба объекта в `entities`
6. NEGATIVE — `entities: []`, никаких PII в тексте
7. Ты не использовал никаких имен или адресов кроме тех, что получил на вход.
8. `expression`: при `rage` — в тексте есть 1–3 мата в контексте (не в сущностях).
